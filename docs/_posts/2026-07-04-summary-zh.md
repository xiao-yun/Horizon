## [PostgreSQL 与 OOM 杀手：为何选择严格内存超额分配](https://www.ubicloud.com/blog/postgresql-and-the-oom-killer-why-we-use-strict-memory-overcommit) ⭐️ 7.0/10

Ubicloud 这家托管 PostgreSQL 服务商发布了一篇博客，解释他们为何禁用 Linux 默认的内存超额分配策略（将 vm.overcommit_memory 设为 2），以避免 OOM killer 意外终止 PostgreSQL 进程。 这对于数据库稳定性至关重要，能防止因内存耗竭导致的服务中断，并为其他生产环境提供了可参考的内存管理实践；同时也引发了关于 Linux 默认内存行为是否合理的社区讨论。 将 vm.overcommit_memory 设为 2（严格超额分配）后，系统仅允许分配不超过物理内存与交换空间之和一定比例的内存，从而杜绝超额分配引发的 OOM 状况；但若没有相应调整过度分配比率，可能导致应用程序 fork 失败。

hackernews · furkansahin · Jul 3, 13:00 · [社区讨论](https://news.ycombinator.com/item?id=48774509)

**背景**: Linux 内核默认启用启发式内存超额分配（overcommit），允许进程申请的内存总量超过物理内存，依赖进程不会同时使用全部内存。当内存真实耗尽时，OOM killer 会强制终止进程以释放内存，这可能意外杀死数据库等关键服务。因此，一些数据库运维场景倾向于禁用超额分配，以预先拒绝非法内存请求来换取稳定性。

## [Iceberg REST API 提议暴露服务端分配的 tableId 以支持访问控制](https://github.com/apache/iceberg/issues/16399) ⭐️ 6.0/10

Apache Iceberg 社区提出功能请求（issue #16399），计划在 LoadTableResponse REST API 响应中暴露服务端分配的资源标识符（如 tableId），以便客户端无需拦截 HTTP 请求即可实现资源级访问控制。 该改进将简化访问控制集成，特别有利于像 Amazon S3 Tables 这样后端分配 tableId 并用于构造 ARN 的系统，使得通过 Iceberg REST 协议联合的下游服务能直接获取凭据和授权。 当前仅提议在加载表（loadTable）的响应中增加 tableId 字段；这一变更属于增量增强，不涉及其他 REST 端点。tableId 是 S3 Tables 等服务端在表创建时生成的唯一标识，用于构建 AWS 资源名称（ARN）。

github · aritragster · May 18, 19:56

**背景**: Apache Iceberg 是一种高性能开放表格式，为数据湖上的大型分析表提供类似 SQL 的表抽象，支持多种引擎并发访问。Iceberg REST Catalog 规范定义了标准化的 RESTful API，允许客户端通过 HTTP 与目录服务交互，替代早期的 Thrift 或特定实现。Amazon S3 Tables 是 AWS 提供的优化分析工作负载的存储服务，它基于 Iceberg 格式并提供 REST 端点，并会给每张表分配一个服务端生成的 tableId，该 ID 用于 IAM 策略中的资源级权限控制。
