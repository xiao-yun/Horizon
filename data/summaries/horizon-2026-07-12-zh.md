## [PgBouncer 吞吐量提升 4 倍：利用 SO_REUSEPORT 与 peering 技术](https://clickhouse.com/blog/pgbouncer-clickhouse-managed-postgres) ⭐️ 7.0/10

## [在 SQLite 中优先使用严格表](https://evanhahn.com/prefer-strict-tables-in-sqlite/) ⭐️ 7.0/10

文章建议在 SQLite 中使用 STRICT 表选项来强制列类型，避免因类型强制转换导致的数据损坏，并基于社区真实案例（如 UUID 被误解析）强调了其必要性。 此举能显著提升数据完整性，防止意外类型转换带来的隐蔽错误，尤其对于多应用共享数据库或依赖数据类型的场景至关重要。 严格表要求每列必须声明数据类型（INT、INTEGER、REAL、TEXT、BLOB、ANY），并会对插入值进行类型转换，但不支持 Date 等类型；迁移现有表需重建。

hackernews · ingve · Jul 11, 17:33 · [社区讨论](https://news.ycombinator.com/item?id=48873940)

**背景**: SQLite 默认采用灵活类型亲缘性（type affinity），允许列存储任意类型值，可能导致如 UUID 被解析为数字等问题。严格表是 3.37.0 版本引入的功能，在 CREATE TABLE 末尾添加 STRICT 关键字即可启用，强制进行类型检查。

## [Apache Iceberg 提议新增 Variant 数据类型以高效存储半结构化数据](https://github.com/apache/iceberg/issues/10392) ⭐️ 7.0/10

Apache Iceberg 社区提交了一份提案（#10392），计划在 Iceberg 的类型系统中加入 Variant 类型。该类型将支持对 JSON、Avro、Parquet 等动态半结构化数据进行高效的二进制编码，以弥补现有存储方式的不足。 这将使查询引擎能够更高效地处理半结构化数据，同时保留源数据的灵活性，对分析日志、遥测和物联网数据等场景有显著的性能提升。Variant 类型正成为数据湖生态系统中的新标准，此举将增强 Iceberg 的竞争力。 Variant 类型基于 Parquet 社区定义的二进制编码，每行数据自包含类型信息，查询时无需解析文本 JSON，从而大幅提升性能。目前该功能尚处提案阶段，未来将纳入 Iceberg v3 规范。

github · sfc-gh-aixu · Apr 30, 12:52

**背景**: Variant 是一种无模式数据类型，能够表示任意嵌套结构的数据，常见于 Databricks、Spark 等系统。在数据湖中，半结构化数据通常以 JSON 字符串形式存储，既牺牲了查询效率，也难以维护演变模式。Apache Iceberg 是一种开放表格式，广泛用于大数据分析，引入 Variant 后可与其他系统更好地互操作，并提升处理灵活数据的能力。

## [Apache Iceberg 提议引入文件格式 API 以统一跨格式特性支持](https://github.com/apache/iceberg/issues/12225) ⭐️ 7.0/10

Apache Iceberg 社区提出 Issue #12225，计划引入一个文件格式 API，旨在解决当前 Avro、Parquet、ORC 等格式之间特性支持不一致的问题，并简化新兴格式的集成。 此举将消除因不同开发者对不同文件格式的专注程度不同而导致的特性碎片化，确保所有格式都能及时支持 V3 规范中的新特性，从而提升 Iceberg 作为数据湖表格式的扩展性和一致性。 该提议主要针对 Iceberg V3 规范引入的新列类型、默认值等特性，这些特性需要文件格式层面的改动，但目前各格式的支持进度不一。API 将提供一个统一接口，使得新增格式或特性时无需重复实现底层细节。

github · pvary · Apr 20, 11:57

**背景**: Apache Iceberg 是一种面向大规模分析表的高性能开源表格式，最初由 Netflix 开发并于 2018 年捐赠给 Apache 基金会。它支持 Spark、Trino、Flink 等多种引擎并发安全地操作同一张表，解决了传统 Hive 表在数据湖中的性能和一致性问题。Iceberg 目前原生支持 Avro、Parquet 和 ORC 三种文件格式，但其新特性往往需要逐个格式适配，导致支持延迟和碎片化。

## [Apache Hudi 提议支持分区软删除功能](https://github.com/apache/hudi/issues/18774) ⭐️ 7.0/10

Apache Hudi 社区提出了一项新提案（#18774），计划为分区删除操作增加软删除支持，允许用户在数据被彻底清理前进行恢复。 这一功能将提升数据安全性，避免误删分区造成永久数据丢失，并为用户提供恢复窗口，符合数据湖治理的最佳实践。 当前该提案仍处于讨论和早期设计阶段（WIP）。现有机制中，delete_partition API 会立即替换分区内所有文件，而清理服务（clean）随后将其永久删除。软删除将允许先标记分区为删除状态，延迟实际的文件移除和元数据清理。

github · kbuci · May 18, 22:43

**背景**: Apache Hudi 是一个开源数据湖平台，在数据湖上提供数据库功能，如增量处理、索引和事务。分区是 Hudi 表的基本组织单元。删除分区通常涉及物理移除文件，而软删除是一种先标记后延迟清理的策略，常用于防止误操作带来的数据丢失。

## [Apache Iceberg 提议在 Spark 中捕获并发出 Parquet 聚合指标](https://github.com/apache/iceberg/issues/16675) ⭐️ 6.0/10

一项针对 Apache Iceberg 的改进提案（#16675），允许在 Spark 写入数据时可选地从 Parquet 文件页脚捕获聚合的物理存储指标，并在提交时通过事件框架发出，不会持久化到表元数据中。 该功能可让用户在不增加表元数据负担的情况下，获取数据文件的物理存储统计（如值计数、空值计数等），以便用于监控、成本优化和性能调优。 指标提取按列块进行，并映射到 Iceberg 字段 ID 以保证模式演进安全；此功能为可选机制，不影响现有写入流程。

github · gtrettenero · Jun 3, 15:58

**背景**: Apache Iceberg 是一种高性能开放表格式，用于管理大规模分析数据集，支持 Spark、Trino 等多引擎安全并发操作。Parquet 是列式存储格式，其文件尾部包含各列块的元数据统计信息。该提案利用 Parquet 的尾部统计，而避免增加 Iceberg 表元数据的复杂性。
