## [Mullvad 推出 VPN 退出 IP 指纹识别缓解措施](https://mullvad.net/en/help/exit-ip-vpn-servers-mitigation-rollout) ⭐️ 7.0/10

Mullvad 正在部署一种新的退出 IP 分配方法，以防止通过 VPN 服务器的退出 IP 进行跨服务器或跨用户的关联指纹识别。 此缓解措施消除了一个细微的隐私威胁，使观察者无法通过切换 VPN 服务器时的退出 IP 模式来追踪用户，提升了注重隐私的用户的匿名性。 此前，退出 IP 基于 WireGuard 密钥确定，可能长时间保持不变；新方法在服务器切换时不再泄露关联信息，Mullvad 建议切换服务器时在应用中注销并重新登录以重新生成密钥。

hackernews · Cider9986 · May 25, 17:45 · [社区讨论](https://news.ycombinator.com/item?id=48269580)

**背景**: VPN 服务器通常拥有多个公共 IP 地址（退出 IP），用户连接时被分配其中一个。如果分配是固定的或可预测的，第三方就可以将同一用户在不同服务器或同一服务器不同时间的活动关联起来。Mullvad 运营的服务器数量相对较少，每台服务器通过多个退出 IP 来分摊用户，以规避网站的 IP 封锁和速率限制。

## [C 头文件编译器扩展的移植性难题与对策](https://lemon.rip/w/6-c-extensions-compilers/) ⭐️ 7.0/10

文章和社区讨论深入剖析了在 C 头文件中处理编译器特定扩展（如`__attribute__`）的困难，指出了仅检查`__GNUC__`等宏的常见陷阱，并提供了具体的替代方案和测试策略，以提升代码在不同编译器间的可移植性。 这对系统程序员意义重大，因为许多开源 C 项目依赖 GCC 扩展，导致在 Windows、BSD 或使用替代编译器（如 TinyCC、SlimCC）时编译失败；采用更健壮的检测方法能显著降低移植成本，促进编译器生态健康发展。 讨论中提到的典型问题：`#if !defined(__GNUC__)`会误判不支持`__attribute__`的编译器；社区建议改用特性检测宏（如检查`__attribute__`是否已定义）或类似 Boost.Config 的细粒度宏，并推荐了 slimcc 的测试脚本和头文件补丁作为参考。

hackernews · xngbuilds · May 25, 14:15 · [社区讨论](https://news.ycombinator.com/item?id=48267126)

**背景**: 编译器特定扩展指 C 标准之外由特定编译器提供的功能，如 GCC 的`__attribute__`用于声明函数属性；许多 C 头文件通过预处理器宏`__GNUC__`判断是否为 GCC 或 Clang，进而使用这些扩展。但并非所有支持该扩展的编译器都定义`__GNUC__`，且同一扩展在不同编译器上行为可能不同，粗暴的宏判断会导致移植性问题。
