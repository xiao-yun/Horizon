## [Voxel Space：1992 年《科曼奇》体素高度图渲染技术详解](https://s-macke.github.io/VoxelSpace/) ⭐️ 8.0/10

技术文章《Voxel Space》详细剖析了 1992 年游戏《科曼奇》中使用的体素高度图渲染算法，并提供了伪代码和交互演示。 该算法在有限硬件上实现了流畅的 3D 地形渲染，是游戏图形史上的里程碑。文章为现代开发者提供了学习经典优化技巧的宝贵资源，展示了算法简洁性带来的卓越视觉效果。 算法本质上是 2.5D 引擎：沿屏幕水平方向逐列扫描，根据高度图计算每条垂直线的起始位置和颜色，一次绘制一列来模拟透视地形。社区指出，它并非真正的三维体素，而是基于二维高度图的棱柱体近似。

hackernews · davikr · May 30, 14:25 · [社区讨论](https://news.ycombinator.com/item?id=48336564)

**背景**: 体素（Voxel）即体积像素，是三维空间的基本单元，常用于表示地形和物体。1992 年，Novalogic 公司的《科曼奇》（Comanche）首次采用基于高度图的渲染技术，借助简单的光栅化操作，在当时主频仅几十 MHz 的 PC 上实现了大面积、细节丰富的户外地形。该技术属于 2.5D 渲染，采样二维高度和颜色数据，每列只绘制一条垂直线，牺牲了真三维自由度以换取性能，是早期计算机图形学的经典案例。

## [Iceberg REST API 提议增加参数以修剪快照和元数据日志](https://github.com/apache/iceberg/issues/15947) ⭐️ 6.0/10

GitHub issue #15947 提议为 Apache Iceberg REST API 的 loadTable 响应增加查询参数，用于修剪 `snapshot-log` 和 `metadata-log` 数组，从而避免这些数组随提交操作无限增长。 随着 Iceberg 表版本累积，`snapshot-log` 和 `metadata-log` 会无限增加，导致 loadTable 响应体积过大、影响性能；该改进可让客户端按需获取历史日志，提升 API 的可用性和效率，对管理大型表元数据的用户尤为重要。 当前 loadTable 响应直接返回完整的 `TableMetadata` 对象，其中 `snapshot-log` 和 `metadata-log` 是每次提交都会追加条目的无界数组；提议新增的查询参数将允许客户端指定返回的日志条目数量或范围，类似对 Snapshots 列表的分页支持。

github · laserninja · Apr 12, 02:33

**背景**: 在 Apache Iceberg 中，`snapshot-log` 记录了每次快照操作的快照 ID 与时间戳，而 `metadata-log` 存储了以往元数据文件的历史信息。这两个日志数组对于审计、时间旅行和回滚等功能至关重要，但随着表上提交次数的增加，它们会持续增长，若不加控制会显著加大元数据传输负担。
